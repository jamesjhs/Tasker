1. Executive Summary

Tasker is a privacy-first, mobile-first web application designed to enable healthcare staff to log and analyse their administrative workload anonymously. It operates as a single-page application (SPA) served by a Node.js/Express back-end, with all data stored locally in a SQLite database file. There are no third-party data processors, no cloud dependencies, and no telemetry.

Key design priorities, in order:

  1. Privacy — usernames are computer-generated word-pairs that cannot be linked to a real person; no names, patient references, or identifying data is ever stored or requested.
  2. Security — CSRF protection, bcrypt password hashing, session-based authentication, field-level AES-256-GCM encryption for notes, account lockout, rate limiting, and strict Content-Security-Policy headers.
  3. Simplicity — one process, one file-based database, zero external services.
  4. Mobile UX — the entire front-end renders inside a single <div id="app"> element and uses vanilla JavaScript with no front-end framework. It installs as a PWA.
  5. Automatic data lifecycle — task records are deleted after 30 days.

The server process is single-threaded Node.js. There is no message queue, no worker pool, and no caching layer. The SQLite database is opened once at startup via a module-level singleton and all queries are synchronous (using better-sqlite3). This design eliminates entire categories of concurrency bugs and is entirely appropriate for a small-team deployment.

6. Server Entry Point — src/server.ts

6.1 Startup Sequence

  1. dotenv/config is imported first as a side-effect, loading .env into process.env.
  2. The Express app object is created and trust proxy is set to 1, enabling correct IP detection behind a reverse proxy.
  3. SSL detection: The process checks whether both SSL_CERT and SSL_KEY files exist on disk. The useHttps boolean controls subsequent TLS-related behaviour.
  4. Helmet applies security headers. The CSP allows 'unsafe-inline' for scripts (required by vanilla-JS inline event handlers in app.js) and cdn.jsdelivr.net (Chart.js). upgradeInsecureRequests is only included when running over HTTPS.
  5. JSON and URL-encoded body parsers are registered with a 1 MB request size limit.
  6. Session middleware is configured with a SQLite-backed store, httpOnly: true, sameSite: 'strict', secure conditional on NODE_ENV=production, maxAge: 30 minutes.
  7. Rate limiters are created: auth (20/15 min) and API (200/min).
  8. The /readyz health-check endpoint is registered before authentication middleware.
  9. express.static serves public/. The sw.js file gets Cache-Control: no-cache and Service-Worker-Allowed: / headers. The index.html shell also gets Cache-Control: no-cache so that HTTP-cache clients always revalidate it and pick up new asset references even when the Service Worker is not in control.
  10. All API route groups are mounted with their respective rate limiters.
  11. Static HTML pages (/policy, /help, /guide) are served by explicit routes.
  12. The catch-all GET /{*path} returns index.html — enabling SPA browser history navigation.
  13. Data retention job: runRetention() deletes tasks older than 30 days on startup and every 24 hours.
  14. getDb() initialises the database connection and runs schema migrations.
  15. The HTTP(S) server starts listening on the configured port.

6.2 Version Exposure

The app's version string is read from package.json at compile time via import { version } from '../package.json' (enabled by resolveJsonModule: true in tsconfig.json). Exposed at /api/version and /readyz.

7. Database Layer — src/db.ts

7.1 Singleton Pattern

The database connection is managed as a module-level singleton: a let _db variable that is null until first use. getDb() is the only way to obtain the connection. This ensures the data/ directory is created before any file is opened, schema initialisation runs exactly once, and a single connection is reused across all requests.

7.2 SQLite Pragmas

  • PRAGMA journal_mode = WAL — Write-Ahead Logging. Readers do not block writers and writers do not block readers, enabling concurrent API requests without serialisation stalls.
  • PRAGMA foreign_keys = ON — Enforces referential integrity (SQLite does not enforce REFERENCES constraints by default).

7.3 Schema Definition (initSchema)

The schema is declared in one idempotent db.exec() call using CREATE TABLE IF NOT EXISTS. The eight tables are:

TablePurpose
user_groupsNamed groupings for users (e.g. "General", "Cardiology"). Controls dropdown visibility.
usersAll accounts: regular users and the admin. Stores bcrypt hash, lockout state, group assignment, activation flags.
settingsKey/value configuration pairs (currently: self_registration, user_invite modes).
tasksThe core data entity. One row per logged task.
dropdown_optionsThe master list of selectable values for category, subcategory, and outcome fields.
group_dropdown_optionsMany-to-many join: which options are visible to which group.
user_dropdown_optionsMany-to-many join: which options an individual user has personally selected.

pending_task_logsTimestamped snapshots of a user's pending task count.

7.4 Automatic Schema Migrations

After the initial schema creation, initSchema reads each table's actual column list using PRAGMA table_info(tableName) and conditionally adds columns introduced in later versions (failed_login_attempts, is_locked, is_approved, pending_activation, user_group_id, assigned_date, etc.). Any existing database from an older version is automatically migrated on the next startup — no manual scripts or downtime required.

7.5 Seed Data

On a fresh database (zero approved dropdown options), default categories, subcategories, and outcomes are inserted atomically. If no user groups exist, a "General" group is created and seeded with all options. Settings default to 'admin_approved'.

7.6 Database Restore

replaceDb(uploadedPath): validates the uploaded path is strictly inside data/restore-tmp/ (path traversal prevention), closes the current database connection, and copies the uploaded file over tasker.db.

8. Field-Level Encryption — src/encrypt.ts

8.1 Purpose and Scope

The notes field on a task is the only free-text field users can write into that is stored in the database. To prevent exposure if the database file is accessed outside the application, notes are encrypted at rest when ENCRYPTION_KEY is configured.

The Send suggestion to developers field in the Notices and Feedback panel is a separate freetext input whose content is transmitted by email to an NHS.net address and is never persisted on the application server. From v1.8.5, this field displays a prominent notice instructing users not to submit any patient, location, or staff-identifiable information.

8.2 Algorithm: AES-256-GCM

  • GCM (Galois/Counter Mode) provides both confidentiality and integrity (any tampering with the ciphertext is detected).
  • The key is a 32-byte value derived from the ENCRYPTION_KEY environment variable (64 hex characters).
  • A fresh 16-byte random Initialisation Vector (IV) is generated per encryption operation via crypto.randomBytes — the same plaintext always produces a different ciphertext.
  • The 16-byte authentication tag is stored and verified before decryption.

8.3 Storage Format

Encrypted values are stored as a prefixed, colon-delimited base64 string:

enc:<IV base64>:<tag base64>:<ciphertext base64>

The enc: prefix distinguishes encrypted from plaintext values, enabling backwards-compatible operation with pre-existing plaintext notes.

8.4 Key Validation

getKey() re-reads ENCRYPTION_KEY from process.env on every call. It validates the 64-hex-character format and returns null if invalid (logging a warning). When null, encryptField passes values through unmodified, and decryptField returns null for any already-encrypted value (preventing silent ciphertext exposure).

9. Middleware — src/middleware/index.ts

9.1 requireAuth

Checks: (1) req.session.userId exists; (2) session activity is within 30 minutes — if not, session is destroyed and 401 Session expired is returned; (3) session.sessionDate matches today — if the session spans midnight, it is destroyed. On success, session.lastActivity is updated.

9.2 requireAdmin

Checks session.isAdmin === true. Returns 403 otherwise. Always stacked on top of requireAuth.

9.3 requirePasswordChange

If session.mustChangePassword is true, blocks all requests except /change-password and /logout. Enforces password change for admin-created and invited users.

9.4 requireActivation

If session.pendingActivation is true, blocks all requests except /logout. Enforces administrator activation of invited accounts.

9.5 validateCsrf

Reads the X-CSRF-Token request header and compares it against session.csrfToken. Returns 403 Invalid CSRF token on mismatch. Required on all POST/PATCH/PUT/DELETE requests.

9.6 logEvent

Non-blocking helper that inserts a row into the events table with an event type string only — no user IDs, no IP addresses, no task IDs. Catches all errors silently.

10. Anonymous Username Generator — src/words.ts

Generates pseudonymous adjective-noun word pairs (e.g. BrightHeron, CalmOak) from lists of 100 adjectives and 100 nouns (10,000 combinations). generateUsername() uses crypto.randomInt(arr.length) (cryptographically secure), checks the result against the users table, and retries up to 200 times on collision. Falls back to appending a 3-digit random number if all 200 attempts collide.

12. Client-Side Application — public/js/app.js

12.1 Architecture: Single-Page Application

A single ~2,750-line JavaScript file with no framework, no JSX, no bundler, no TypeScript. The HTML shell contains a single <div id="app">. View transitions are synchronous DOM replacements: app().innerHTML = renderXxx().

12.2 Application State

A global state object holds all runtime state: csrfToken, user, registrationConfig, appStats, activeTask, timer intervals, interruptStart, dropdowns, taskForm, editTask, charts, pendingTaskLog, recentHandledCount.

12.3 API Helper (api function)

A thin fetch wrapper that: adds X-CSRF-Token and Content-Type headers; handles 401 (redirects to login), 429 (rate-limit message with wait time), and non-OK responses (extracts error from JSON body).

12.4 Initialisation (init)

Called on DOMContentLoaded: register SW → check asset version (cache purge + reload on version bump) → fetch CSRF token → call /api/auth/me → if authenticated, load dropdowns + active task + start activity tracking (which also starts the background version poll) + render view; if not, render login.

12.5 View Navigation & Browser History

Every view function calls pushHistory(viewName) or replaceHistory(viewName), storing the view name in window.history.state. A popstate listener maps state names to render functions via HISTORY_RENDER_MAP, with auth guards preventing access to protected views when logged out.

12.6 Session Inactivity Tracking

Activity events (click, touchstart, keydown, scroll) update a localStorage.tasker_last_active timestamp via updateLastActive(). A 60-second interval also calls updateLastActive() to keep the timestamp current while the user is passively reading. A separate 60-second interval calls checkClientInactivity(), which implements a two-stage response:

  • Warning stage (5–30 min idle). A slim amber strip (.inactivity-warn) is appended to document.body showing "App last used: HH:MM". It has pointer-events: none so the app remains fully interactive underneath. Any qualifying user event calls updateLastActive(), which dismisses the strip and resets the timestamp.
  • Expiry stage (≥30 min idle). forceSessionExpiry() is called: activity tracking is stopped, the server logout endpoint is called, client state is cleared, SW caches are deleted, and the inactivity screen is rendered. "Log in again" performs location.reload(), guaranteeing a fresh CSRF token and clean client state.
  • Interaction-triggered expiry. updateLastActive() also checks the elapsed time; if an event fires after the 30-minute deadline has passed, it calls forceSessionExpiry() immediately rather than silently resetting the clock.
  • Guard flag. A _sessionExpiryInProgress boolean prevents forceSessionExpiry() from being entered a second time if a visibilitychange event fires between stopActivityTracking() and state.user = null.

The visibilitychange event triggers an immediate checkClientInactivity() call when the tab becomes visible, and also calls checkAssetVersion() (throttled to once per 60 s) so that users returning to a backgrounded PWA always receive the latest code. If a task was active during inactivity, the gap is automatically recorded as an interruption via checkInactivityInterruption().

12.7 Combobox Component

A custom searchable dropdown with: trigger button, search input, options list, and "Add new" row. Keyboard-navigable (arrow keys, Enter, Escape). Only one open at a time. The filterCombo function re-renders the options list on each keystroke.

12.8 Chart Rendering

renderChart(canvasId, type, labels, datasets, options) wraps new Chart(ctx, {...}). Instances stored in state.charts and destroyed by clearCharts() before any view transition. Charts are grouped into four logical sections: overview (category time doughnut, My Group vs Personal bar, outcome distribution doughnut, outcome breakdown by category stacked bar), task-type & duration (avg duration by category, tasks by type, avg duration by task type, task types by source group stacked bar), flags (flag distribution, flags by source group stacked bar), and temporal patterns (hour-of-day, day-of-week, task type patterns by DOW stacked bar, tasks/time/regression trend line, interruptions trend line, assignment-to-action lag histogram). New charts are only rendered when sufficient cross-tab data exists.

12.9 XSS Prevention

All user-supplied data rendered into HTML is passed through esc() (HTML-entity encoding of &, <, >, ", '). A secondary helper safeId() strips all non-alphanumeric/hyphen/underscore characters from strings used as DOM element ID suffixes.

13. Progressive Web App (PWA) Layer

13.1 Web App Manifest (public/manifest.json)

display: standalone (removes browser chrome on home screen launch), theme_color: #1a56db, three icon sizes (SVG, 192px PNG, 512px PNG).

13.2 Service Worker (public/sw.js)

Cache name versioning (tasker-v4). Bumping the constant in a new release triggers cache invalidation for all users.

  • Install: Pre-caches static assets using Promise.allSettled (one failure doesn't abort install). Calls self.skipWaiting().
  • Activate: Deletes stale caches. Calls self.clients.claim().
  • Fetch strategy — Network-first for /api/*: Always attempts network; returns 503 JSON on failure.
  • Fetch strategy — Cache-first for everything else: Serves from cache; falls back to network (and caches the response). On failure, returns cached / for navigation requests.

The sw.js file is served with Cache-Control: no-cache, ensuring the browser always revalidates it.

13.3 Three-Layer Cache-Update Strategy

Mobile PWA wrappers (standalone home-screen installs, TWA/WKWebView shells) cannot perform a hard cache refresh. Three complementary mechanisms ensure users always receive updated code after a server deployment:

  1. Foreground-resume check (Option 1). The visibilitychange event handler calls checkAssetVersion() every time the app returns to the foreground, throttled to once per 60 seconds. If the server reports a different version than the one stored in localStorage, all Service Worker caches are deleted, the SW is unregistered, and window.location.reload() is called. The next load fetches fresh assets from the network.
  2. Background version poll (Option 2). While a user is logged in, startActivityTracking() starts a setInterval that calls checkAssetVersion() every 5 minutes (state.versionPollInterval). This catches the case where a user leaves the app open without backgrounding it (e.g. a shared device left on-screen all day). The poll shares the same 60-second debounce timestamp as the foreground-resume check, so both paths are always in sync. The interval is cleared by stopActivityTracking() on logout or session expiry.
  3. HTTP no-cache on index.html (Option 3). The express.static setHeaders callback sets Cache-Control: no-cache on the HTML shell. This instructs the browser's HTTP cache to always revalidate index.html with a conditional request (ETag / If-None-Match) before using a cached copy — even in environments where the Service Worker is not in control (first visit, after SW unregistration, browsers that do not support SW). A 304 Not Modified response is extremely cheap, and the browser can still serve the cached copy when nothing has changed.

14. Security Design

14.1 Authentication

  • Passwords hashed with bcrypt cost factor 12 (~250ms/hash).
  • Sessions use a server-side SQLite store; the client holds only a signed opaque cookie.
  • Sessions expire after 30 minutes of inactivity (server-side and client-side). A 5-minute inactivity warning overlay is shown client-side before the timeout fires.
  • Sessions are terminated at midnight (server-side sessionDate check).

14.2 CSRF Protection

Token stored server-side in the session. Client sends it as a custom X-CSRF-Token header. Immune to cross-origin attacks because: cookies are SameSite=strict; the token cannot be read by a cross-origin script; custom headers cannot be set by HTML forms.

14.3 Content Security Policy

Blocks inline script (with 'unsafe-inline' exception), external scripts except Chart.js CDN, external images, external styles, external connections except Chart.js CDN.

14.4 Rate Limiting

  • Auth limiter: 20 requests per 15 minutes per IP, skipSuccessfulRequests: true.
  • API limiter: 200 requests per minute per IP.

RFC 9110 RateLimit-* headers returned so the client can display informative wait-time messages.

14.5 Account Lockout

After 3 consecutive failed login attempts, is_locked = 1 is set. Unlocking requires explicit administrator action. Remaining attempt count communicated in error messages.

14.6 Path Traversal Prevention

The database restore endpoint validates that the resolved uploaded file path starts with path.resolve(RESTORE_DIR) + path.sep, preventing path construction attacks.

14.7 Input Validation

  • Dropdown ALLOWED constant whitelists three permitted field names.
  • Notes: 2,000-character server-side limit.
  • Dropdown values: 100-character limit.
  • Group names: 80-character limit.
  • Pending counts: integer, 0–9,999.
  • All values rendered into client HTML passed through esc().

    ├─ (auto)           → is_approved=1 → user logs in immediately

Existing user → "Invite a User" → POST /api/auth/invite

17. Task Lifecycle & Workflows

Only one in_progress task can exist per user at any time (enforced server-side).

17.1 Normal Task Flow

User taps "Log Task"
→ selects type (duty/personal), category, subcategory, assigned date
→ POST /api/tasks/start → status='in_progress'
→ Live timer display

[Optional] User taps "Interrupted"
→ showInterruptModal

18. Analytics Engine

18.1 Session vs History

The session view shows data for the current calendar day (anchored to session.sessionDate at login — prevents cross-midnight data leakage). The history view queries the past 30 days (maximum retention window).

18.2 Linear Regression Algorithm

Ordinary least squares on daily task counts, pure JavaScript:

slope     = Σ(xi - x̄)(yi - ȳ) / Σ(xi - x̄)²
intercept = ȳ - slope × x̄
R²        = 1 - SS_res / SS_tot

Calculated when ≥ 3 distinct dates exist. Displayed as e.g. +0.3 tasks/day (R²=0.85, increasing).

18.3 Client-Side Insight Generation

buildInsights(summary) produces human-readable insight strings: busiest hour, busiest day of week, most time-consuming category, most interrupted category, average task duration classification, interruption rate (% of tasks with ≥1 interruption), average assignment-to-action lag, longest task type by average duration, and top flag–source-group combination.

18.4 Cross-Tab Charts (v1.8.7)

Six additional charts appear when sufficient data is present, grouped logically with their related base charts:

  • Outcome Breakdown by Category — stacked bar; outcome distribution per source group (shown when >1 category)
  • Avg Duration by Task Type — horizontal bar; average minutes per subcategory
  • Task Types by Source Group — stacked bar; task-type composition per category (shown when >1 category or any category has >1 type)
  • Flags by Source Group — horizontal stacked bar; which source groups generate which flags
  • Task Type Patterns by Day of Week — stacked bar; temporal pattern of task types (shown when ≥2 distinct types present)
  • Days from Assignment to Action — histogram of lag in days between assigned_date and start_time, with avg/median/range displayed above the chart

18.5 Personal Assignment-Day Charts (v1.8.8)

Two further stacked column charts show the day-of-week distribution of Personal task assignments only (tasks with is_duty = 0 and a populated assigned_date). Each is displayed when at least one Personal task with an assigned date is present in the current data set:

  • Personal Tasks by Day Assigned — by Task Origin — stacked column; each column represents a day of the week; each stack segment represents a task origin (category), drawn using the same COLORS palette and renderChart helper as all other charts
  • Personal Tasks by Day Assigned — by Task Type — stacked column; same axes but stack segments represent task types (subcategory)

The underlying data is produced by buildSummary fields byDowPersonalByCategory and byDowPersonalBySubcategory respectively (see §18.2).

19. Admin Workflows

19.1 Admin Panel Initialisation

renderAdmin() issues 8 parallel API requests in one Promise.all (stats, users, dropdown options, settings, pending users, awaiting-activation users, user groups, pending group proposals). All data is passed to renderAdminContent() in a single pass.

19.2 Group Dropdown Management

Managed via a modal dialog. PUT /api/admin/user-groups/:id/dropdowns atomically replaces the entire group_dropdown_options set inside a transaction. Users with pre-seeded personal options are not automatically updated.

19.3 Database Backup & Restore

Backup streams the raw SQLite file (safe during operation in WAL mode). Restore is destructive and immediate: closes the DB handle, overwrites the file, re-opens on the next request.

20. Testing Architecture

Located in src/__tests__/. Framework: Jest + ts-jest + supertest.

  • Test database isolation: TASKER_DB_PATH env var set to a temp file. closeDb() called in afterAll hooks.
  • Test app helper (helpers/testApp.ts): Creates a configured Express app without starting a TCP listener. Supertest makes direct calls to the app object.
  • Security test coverage: CSRF enforcement, rate limiting, authentication gates, admin gates, password complexity, account lockout behaviour.

Run all tests: npm test.

22. Data Retention & Privacy

22.1 Automatic Task Deletion

DELETE FROM tasks WHERE created_at < datetime('now', '-30 days')

Runs on startup and every 24 hours. Result (rows deleted) is logged to console if non-zero.

22.2 User-Initiated Deletion

Users can delete their account from the Settings page. Requires typing username + password. Cascades to delete all associated task, option, and log data.

22.3 Privacy by Design

  • Usernames are computer-generated and cannot be linked to real identities.
  • The events table stores only event type + timestamp — never user IDs.
  • A Privacy Splash is shown on every login, reinforcing the prohibition on entering patient-identifiable data.
  • Task notes (the only free-text field stored in the database) are encrypted at rest when configured.
  • The Send suggestion to developers field is the only other freetext input; submitted text is emailed to an NHS.net address and never persisted on the server. An on-screen notice (from v1.8.5) explicitly warns users not to include patient, location, or staff-identifiable information.
  • Aggregate statistics are shown publicly on the login page; individual data is always scoped to the authenticated user.

23. Known Design Decisions & Constraints

DecisionRationale
No front-end frameworkEliminates build complexity; entire front-end is a single deployable file. Trade-off: more manual DOM management; inline event handlers require 'unsafe-inline' in CSP.
better-sqlite3 (synchronous)Eliminates async complexity for a read-heavy, single-process workload. Would not scale beyond ~100 concurrent users without switching to an async driver.
Separate sessions.db filePrevents session-churn fragmentation of the main database. sessions.db can be discarded without data loss.
Notes encrypted; category/subcategory/outcome not encryptedOnly the free-text field is likely to contain sensitive narrative content. Controlled-vocabulary fields are used in SQL aggregations and analytics — encrypting them would prevent these queries.
Mobile-only Excel exportPrevents accidental download of task data to a shared desktop computer. Mobile devices are typically personal devices.
Session terminated at midnightEnsures task start/end times are always on the same calendar day as the session, keeping session analytics coherent.
Three-tier dropdown resolutionAllows the system to function for users without a group (global fallback) while letting groups and individuals see curated subsets.
mfa_enabled column reservedColumn exists in the schema but MFA logic is not yet implemented — reserved for a future release.
Linear regression computed server-sideKeeps the calculation deterministic, auditable, and independent of client capabilities. No additional client-side library needed.
