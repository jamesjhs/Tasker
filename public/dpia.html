1. Project Description

1.1 Purpose of the Application

Tasker is a lightweight, mobile-first, single-page web application that enables NHS healthcare staff to record anonymous task-workload metadata in real time. Its primary use case is personal productivity and workload analysis, not patient care management.

1.2 Scope of This DPIA

This assessment covers the following functions of Tasker:

  • User registration and account management (username generation, password management, admin activation)
  • Authentication: login, session management, password change, logout
  • Task logging: creating, timing, pausing (interrupt), and completing tasks with metadata (category, subcategory, outcome, and structured task flags)
  • Task history: viewing, editing, and exporting tasks (CSV / Excel)
  • Analytics: personal workload charts and summary statistics
  • Admin functions: user list management, account activation/deletion, aggregate statistics
  • Service Worker (offline caching) and automatic data retention (30-day deletion)

1.3 What This Application Is Not

⚠️ Tasker is not a clinical system. It must never be used to record patient information, clinical outcomes, or any personally identifiable data relating to patients, staff, or third parties. It is not connected to any NHS system, EPR, or shared network infrastructure.

3. Necessity & Proportionality

3.1 Is the Processing Necessary?

The processing of anonymised task metadata is the minimal data required to deliver workload logging functionality. No patient or staff identifiers are required or collected; anonymous usernames are computer-generated and not linked to any real identity.

3.2 Is the Processing Proportionate?

Data collected is limited to task type, timestamps, category, subcategory, outcome, and structured task flags (predefined list). All records are automatically and permanently deleted after 30 days. The scope is proportionate to the legitimate interest in supporting staff workload analysis.

3.3 Privacy by Design Measures

  • Anonymous usernames — computer-generated two-word identifiers with no link to real identity
  • Passwords hashed with bcrypt (cost factor 12) — irreversible; not readable by anyone
  • Free-text notes removed; structured task flags used instead
  • Automatic 30-day data deletion runs daily
  • No email addresses, phone numbers, or real names are stored
  • Sessions expire after 30 minutes of inactivity and cannot span midnight
  • No third-party analytics, no telemetry, no external data sharing

4.2 Data Storage

  • Application database: SQLite file on Administrator's server. Contains anonymised task records, anonymous usernames, and bcrypt password hashes.
  • Session database: Separate SQLite file holding active session tokens only. Auto-expires.
  • No cloud storage: No data is stored in cloud services, shared drives, or NHS systems.
  • No logs containing personal data: Server-side logs record operational events (e.g. server start, retention runs) and do not log user content or identifiers beyond session IDs managed by the session library.

5. Lawful Basis & Special Category Data

5.1 Lawful Basis (UK GDPR Article 6)

Legitimate Interests (Article 6(1)(f)) — The Administrator has a legitimate interest in understanding and recording the workload activity of the team. The processing uses anonymised data which is proportionate and does not override the rights and freedoms of data subjects, given the absence of any identifying information.

Alternatively, where individual staff members voluntarily self-register and use the tool for their own workload analysis, Consent (Article 6(1)(a)) may be the appropriate lawful basis. The Administrator should confirm which basis applies in their context and document this accordingly.

5.2 Special Category Data (UK GDPR Article 9)

✅ No special category data is processed. The application is expressly designed to exclude any health, biometric, racial, religious, or other special category data. The policy strictly prohibits users from entering such data.

From v1.8.1 the free-text notes field has been removed entirely. Task annotations use only structured, administrator-defined flag options. Dropdown and flag suggestions from users are transmitted by email and never stored on the server, further reducing residual data risk.

From v1.8.5 the Send suggestion to developers feature is accompanied by an explicit on-screen notice instructing users not to include any patient, location, or staff-identifiable information. Submitted text is sent directly to an NHS.net email address and is not stored on the application server.

9. Residual Risks & Recommendations

⚠️ Action Required (R1): The medium residual risk (accidental PII entry) cannot be fully mitigated by technical controls alone and requires organisational safeguards.
  1. Communicate acceptable use before staff on-boarding — Provide all users with a written summary of what data may and may not be entered, confirming that no free-text data entry is available except for the Send suggestion to developers field, which must not contain patient, location, or staff-identifiable information. Include a reference to this DPIA.
  2. Obtain a copy of organisational IG sign-off — The Administrator must share this DPIA with their NHS organisation's DPO or IG lead and obtain documented approval before clinical staff use the service.
  3. Consider self-hosting Chart.js — In high-security environments, self-host the Chart.js library to eliminate the CDN dependency and reduce supply-chain risk (R9).
  4. Configure SMTP settings — The Administrator must configure the SMTP settings in the admin panel so that dropdown and flag suggestions are emailed rather than logged, and so that the Send suggestion to developers feature routes freetext to the designated NHS.net address. This is recommended before opening the application to users.
  5. Log and monitor access anomalies — The Administrator should review server logs periodically for unusual login patterns and investigate any account that appears to be sharing credentials.
  6. Review this DPIA annually — Or immediately following any material change to the application's functionality, data flows, or deployment environment.
  7. Incident response procedure — Define a procedure for responding to a potential data breach (e.g. accidental entry of patient data). This should include: identify, contain (delete account/data), assess, notify DPO if required by UK GDPR Article 33.
